# cv-healthcheck Codebase Review
**Date:** 2026-05-20  
**Reviewer:** Claude Code  
**Scope:** Full codebase read-only review

## Summary

The codebase is in a healthy mid-refactor state: the planned `quickhc/` registry architecture (`models.py`, `registry.py`, `report_service.py`) is fully in place and actively used, the route layer is thin and delegates well to services, and the CSS design-token system is disciplined. The two largest structural debts are (1) the `shared.py` god-module in the route layer, which imports from virtually every domain service and hosts four large preview-data helpers that belong in a future `overview_service.py`, and (2) `license_summary/service.py` taking a hard dependency on `security_assessment.registry.SecurityAssessmentArtifactRegistry` — a naming collision that will cause confusion and a real circular-tendency risk as the codebase grows. The planned `renderers/` layer and `overview_service.py` are not yet present; the five per-tile preview-data helpers in `shared.py` are the exact code that should move there. Everything else is polish rather than structural risk.

## Findings

### Critical (fix before next customer engagement)

**`src/cvhealthcheck/license_summary/service.py:15,65,243,311,358`** · `LicenseSummaryService` imports and instantiates `SecurityAssessmentArtifactRegistry` directly from `cvhealthcheck.security_assessment.registry`. The registry is domain-agnostic (it stores any artifact type via the `artifact_type` column), but the class is named `SecurityAssessmentArtifactRegistry`. Any future refactor of `security_assessment/registry.py` — including renaming the class to `ArtifactRegistry` as planned — breaks `license_summary/service.py` silently. This is also a conceptual coupling: the License Summary service owns its own SQLite DB at `data/imports/license_summary/artifact_registry.sqlite3` yet uses a class named after the competing domain. **Fix:** Rename `SecurityAssessmentArtifactRegistry` to `ArtifactRegistry` (or `GenericArtifactRegistry`) in `security_assessment/registry.py` and update both consumers, or extract the class to a shared `cvhealthcheck.artifact_registry` module before the rename causes more confusion.

**`src/cvhealthcheck/quickhc/report_service.py:139,176,213,310,385,417,448,467`** · All `detail_url` values are hardcoded string literals (`"/quick-hc/security-assessment"`, `"/metrics/client-growth"`, etc.) inside `_build_*` section methods. The `TileDefinition.detail_endpoint` field already holds the Flask endpoint name (e.g. `"main.quick_hc_security_assessment"`). When any route path changes, the report will silently send customers to dead links. **Fix:** Pass `url_for(tile.detail_endpoint)` from the route layer into `build_report()`, or have `_build_*` methods reference the registry's `detail_endpoint` and let the caller resolve the URL. This cannot be done purely inside `report_service.py` (no Flask context there), so the route `quick_hc_report` should resolve and inject the URLs before calling `build_report()`.

### Important (address in next work session)

**`src/cvhealthcheck/web/routes/shared.py`** · `shared.py` is a god module: it defines the blueprint, `login_required`, and all session helpers, but also imports from 15+ service modules and houses five large preview-data helpers (`_license_summary_quick_hc`, `_commcell_quick_hc`, `_client_growth_quick_hc`, `_capacity_license_quick_hc`, plus `security_assessment_quick_hc` re-exported from `reportsplus.security_assessment`) and four chart-builder functions (`_client_count_chart`, `_client_growth_chart`, `_capacity_license_chart`, `_client_growth_detail_chart`). These five preview helpers are precisely the code that should move to the planned `quickhc/overview_service.py`. The chart helpers belong in a metrics presentation layer, not in the shared route bootstrap. Every route module importing `from .shared import ...` pulls in all of this, making test isolation impossible. **Fix:** (1) Extract blueprint/session helpers into a minimal `web/routes/_core.py` or keep in `shared.py` as-is; (2) move the five `_*_quick_hc()` helpers to `quickhc/overview_service.py`; (3) move chart builders to `metrics/charts.py` or similar.

**`src/cvhealthcheck/web/routes/security_assessment.py:34`** · The `reportsplus_security_assessment` route does a local deferred import (`from . import main as main_routes`) inside the view function to avoid a circular import, then calls `main_routes.is_authenticated()` and `main_routes.extract_security_assessment()`. Both functions are already available directly on `shared.py` (they are exported there). The deferred import is a symptom of `main.py` doing the blueprint registration by side-effect; the real fix is the god-module split above. For now, the route should import `is_authenticated` and `extract_security_assessment` from `shared` directly instead of re-importing `main` at call time.

**`src/cvhealthcheck/web/routes/reportsplus.py:110-118` and `143-172` and `173-203`** · `reportsplus_report_metrics`, `reportsplus_health_candidates`, and `reportsplus_execution_validation` are not protected by `@login_required`. The first reads metric data from disk; the other two read local catalog JSON. These are internal/dev routes that should require authentication for consistency. Current behavior: anyone who can reach the app can browse local catalog and metric inventory data without logging in.

**`src/cvhealthcheck/web/routes/metrics.py:17-56`** · All three metrics routes (`/metrics/client-count`, `/metrics/client-growth`, `/metrics/capacity-license`) have no `@login_required`. These pages are reachable without a session and display potentially sensitive operational data from disk.

**`src/cvhealthcheck/reportsplus/security_assessment.py:47`** · `extract_security_assessment()` calls `persist_security_assessment_artifact(normalized)` unconditionally without checking the HTTP status first. If the API returns 401, the function persists an artifact with no findings to the registry and then proceeds to `load_active_security_assessment_artifact()`. The 401 guard is done a few lines later in `SecurityAssessmentService.collect_from_rest()`, but the damage is already written to disk. **Fix:** Guard `persist_security_assessment_artifact` with a check that `normalized.get("source", {}).get("http_status") != 401` before persisting.

**`src/cvhealthcheck/config.py:26`** · `project_root()` returns `Path.cwd()`. All catalog paths (`CATALOG_DIR = Path("data/catalog")`, `SECURITY_ASSESSMENT_IMPORTS_DIR = Path("data/imports/security_assessment")`, etc.) are CWD-relative `Path` literals. This means the app silently writes and reads from whatever the working directory is at startup. If Flask is started from a different directory, all artifacts land in the wrong place with no error. `start.sh` mitigates this by `cd`-ing to the repo root, but it is a fragile convention. **Fix:** Anchor catalog paths to `Path(__file__).resolve().parents[N]` or to an explicit `CV_DATA_DIR` env var.

**`src/cvhealthcheck/quickhc/report_service.py:411`** · `_build_client_growth_section()` returns `"message": "Not collected yet"` even in the `available=True` branch (line 411). This stale message key is probably a copy-paste oversight from the `available=False` branch and could render "Not collected yet" in the customer report if the template references `report.client_growth.message` when the section is available.

### Minor / Polish

**`src/cvhealthcheck/web/app.py:18`** · `app = create_app()` at module level means importing the module creates the app unconditionally. This is a known Flask convention but makes testing harder. Consider removing the module-level singleton so tests control app creation.

**`src/cvhealthcheck/web/routes/quick_hc.py:36`** · `quick_hc()` passes the raw `QUICK_HC_TILE_BY_ID` dict to the template. `quick_hc.html` then does `quick_hc_tiles["environment"]` etc. with string literals. Once the tile registry has a stable ordered API, it would be cleaner to pass an ordered list rather than a dict and avoid string-keyed lookups in Jinja.

**`src/cvhealthcheck/quickhc/registry.py:188-204`** · `report_subsection_options()` and `report_overview_default_selection_ids()` are module-level functions rather than top-level constants. `report_service.py` calls them at import time to assign `REPORT_SUBSECTION_OPTIONS` and `REPORT_OVERVIEW_DEFAULT_SELECTION_IDS`. This is fine, but the function/constant split is asymmetric with the other `QUICK_HC_*` constants. Consider making them pure constants if the registry is always static.

**`src/cvhealthcheck/license_summary/service.py:210-211`** · The `resolved_import_method` logic for `"rest"` source type incorrectly falls through to the `else "rest"` branch: `"upload" if str(artifact.get("source_type") or "").lower() in {"html", "csv", "rest"} else "rest"`. The set includes `"rest"`, so REST-sourced artifacts get `import_method="upload"`. This is the opposite of the intended logic. Compare with `security_assessment/service.py:255`, which has the correct set `{"html", "csv"}`.

**`src/cvhealthcheck/web/templates/quick_hc_report.html:4-17`** · The `usage_summary_cell` macro is defined inline in `quick_hc_report.html`. It is likely to be needed in other report output pages as the tile count grows. This should be extracted to a dedicated partial (e.g. `partials/usage_summary_cell.html`).

**`src/cvhealthcheck/web/routes/shared.py:99-103`** · `_auth_failure_redirect()` checks `result.status_code` via `getattr`. This is inconsistent with the rest of the codebase, which uses `result.status_code` directly on `ApiResult` objects. The `getattr` pattern masks type errors.

**`src/cvhealthcheck/web/templates/base.html:84`** · The fallback URL `/quick-hc` in the Back button JS is a hardcoded string (`window.location.href = '/quick-hc'`). If the Quick HC route path ever changes, this silently breaks. Consider rendering `{{ url_for('main.quick_hc') }}` into a `data-fallback-href` attribute and reading it in JS.

**Chart.js loaded via CDN** (`quick_hc_report.html:402`, `metric_detail.html:77`) · No integrity SRI hash is applied to the CDN `<script>` tags. For an internal tool this is low risk, but adding `integrity="sha384-..."` and `crossorigin="anonymous"` is a one-line fix.

## Template Audit

### Repeated patterns not yet extracted to partials

**Flash message rendering** — The `[{"category": ..., "message": ...} for category, text in get_flashed_messages(...)]` pattern and its Jinja rendering loop appears independently in:
- `quick_hc.py` route (built in Python, passed as `flashes=`)
- `quick_hc_security_assessment.html` (rendered via `flashes` var)
- `license_summary.html` (same pattern)

A `partials/flash_messages.html` macro would DRY this up.

**Section-level `{% if <thing>.available and <thing>.has_content %}... {% else %} <p>{{ <thing>.message }}</p> {% endif %}` pattern** — used five times in `quick_hc_report.html` (once per subject tile). This could become a `report_section_wrapper` macro, though it may not be worth extracting until the tile count grows.

**Preview dispatching via `{% if *_preview_section == "..." %}`** — all five preview partials use an `if/elif` chain dispatching on a Jinja-scoped context variable (`security_preview_section`, `license_preview_section`, etc.). This is a workable approximation of the planned renderer dispatch, but it means each new section sub-type requires editing the partial. When `renderers/` lands, each `if/elif` branch should become a separate partial and the dispatch should be driven by the registry's `preview_renderer` field rather than a passed context variable.

### Logic-heavy Jinja blocks

**`quick_hc.html:106`** — `{% set starting_total = (client_growth.latest_total_clients or 0) - ((client_growth.latest_added or 0) - (client_growth.latest_removed or 0)) %}` — arithmetic in the template. This derived value should be computed in `_client_growth_quick_hc()` in `shared.py` and passed as `starting_total` or incorporated into the preview helper.

**`quick_hc.html:58`** — `(security_assessment.summary.counters["Total checks"] ~ " checks") if security_assessment.exists and security_assessment.summary else "Overview"` — this compound conditional string is assembled in Jinja. This hint text should be produced by the preview helper.

**`quick_hc_report.html:24-26`** — Three `{% set ... %}` variables filter out sentinel strings (`"unknown_customer"`, `"unknown_commcell"`, `"Unknown CommCell"`) inline. These sentinel values are defined as constants in `security_assessment/service.py` but are not accessible from the template. The filtering should happen in `_build_environment()` in `report_service.py`.

## Documentation Gaps

- **README.md** is detailed and mostly accurate. One inaccuracy: the URL table at the bottom (starting line 749) still lists routes like `/reportsplus/security-assessment` and internal development pages as primary URLs, mixing them with the Quick HC surface without indicating which are customer-facing vs internal. The Quick HC section (starting around line 37) is up to date.
- **No docstrings on service classes or public methods.** `QuickHcReportService`, `SecurityAssessmentService`, `LicenseSummaryService`, and their `build_report`/`_build_*`/`_filter_*` methods have no docstrings. Given the complexity of `_build_capacity_license_section()` and `_filter_license_summary()`, at minimum a one-line summary per method would help the next developer.
- **`_filter_*` methods in `report_service.py`** — what each does:
  - `_filter_environment(section, requested, selection_ids)` — gates `show_metadata` on `ENVIRONMENT_METADATA_SECTION_ID` being in `selection_ids`; returns `rows` only when `show_metadata` is true.
  - `_filter_security_assessment(section, requested, selection_ids)` — gates `show_summary`, `show_highlights`, `show_all_findings` on respective section IDs; on unavailable artifacts returns empty lists for all display fields.
  - `_filter_license_summary(section, requested, selection_ids)` — gates four sub-views (`show_summary`, `show_workload_sections`, `show_other_licenses`, `show_agent_feature_licenses`); on unavailable returns empty lists.
  - `_filter_client_growth(section, requested, selection_ids)` — gates `show_summary`, `show_chart`, `show_monthly_table`; strips `chart` and `rows` when not selected.
  - `_filter_capacity_license(section, requested, selection_ids)` — gates `show_summary`, `show_table`; strips `latest_rows` when not selected.
- **`overview_service.py` is planned but not present.** The five `_*_quick_hc()` helpers in `shared.py` are its intended content. Until that module exists, developers may not know where to put new preview-data logic.
- **`renderers/` directory is planned but not present.** The `preview_renderer` and `report_renderer` fields on `TileDefinition` and `SectionDefinition` hold logical renderer names (e.g. `"commcell_preview"`, `"environment_report"`) but these names are not yet wired to anything. A `RENDERER_MAP` or `renderers/` package is the intended next step per README. New contributors may attempt to use these names as template paths.
- **`pyproject.toml`** lists `beautifulsoup4`, `Flask`, and `requests` as dependencies but omits `werkzeug` (used directly in `security_assessment/service.py` and `license_summary/service.py` for `secure_filename`). Werkzeug is a Flask transitive dep, so it will always be present, but the explicit usage deserves documentation or a comment.
